01

Model mental: CLAUDE.md = konstitusi

// kenapa file ini lebih penting dari prompt mana pun

Claude Code menarik CLAUDE.md ke konteks otomatis di setiap sesi — ia jadi sumber kebenaran tentang cara repo-mu bekerja, tanpa kamu menjelaskan ulang. Inilah alasan kontacmu harus ramping: tiap baris memakan context window yang sama dengan kode.

Konsekuensi praktis: kamu tidak perlu menempel aturan layer atau hardening ke tiap prompt. Cukup rujuk: "ikuti kontrak". Yang kamu ketik di terminal adalah tugas, bukan aturan — aturan sudah hidup di file.

Tips dari praktik: aturan bergaya "Jangan ..." kadang terlewat. Kalau Claude Code mulai mengabaikan satu larangan, ubah jadi bentuk "Lebih baik X daripada Y". Mis. ganti "Jangan query di controller" → "Controller hanya memanggil Action lalu kembalikan Resource" (kontrakmu sudah pakai gaya ini — pertahankan).

03

Jembatan & HANDOVER.md

// kontinuitas antar perangkat & antar sesi — file yang bicara, bukan ingatan

Alur yang kamu pakai sudah benar: rancang & debat arsitektur di chat (HP), lalu bawa hasilnya ke Claude Code (laptop). Yang membuat ini mulus adalah file sebagai jembatan — bukan menyalin percakapan.

Di HP (claude.ai)

Putuskan kontrak & alur. Output = CLAUDE.md + docs/*. Tidak ada kode dieksekusi — murni keputusan desain.

Di laptop (Claude Code)

Commit file itu. Claude Code membaca kontrak, lalu kamu kasih tugas per layer. Eksekusi & test di sini.

Tapi menjembatani perangkat saja tidak cukup — kamu juga menjembatani sesi. Sesi Claude Code berikutnya tidak ingat apa pun. Di sinilah HANDOVER.md bekerja: CLAUDE.md menjawab "bagaimana repo ini bekerja" (jarang berubah); HANDOVER.md menjawab "kita sedang di mana" (berubah tiap sesi). Sesi/perangkat baru membaca HANDOVER.md setelah CLAUDE.md untuk langsung nyambung.

04

Rencana build — 11 fase + prompt siap-tempel

// bangun dari data ke atas · satu fase = satu sesi · tiap prompt bisa di-copy

Kontrak mengunci tiga layer: Data → Business → HTTP. Bangun mengikuti arah itu, satu fase sampai test hijau sebelum lanjut. Jangan minta "buatkan semua endpoint" sekaligus — konteks meledak, hardening terlewat. Tiap prompt di bawah mengasumsikan CLAUDE.md sudah ke-load; tap untuk buka & copy.

Fase 0Scaffold proyeksetup
prompt
Setup proyek baru sesuai Stack di CLAUDE.md: Laravel 12, PHP 8.3
strict_types, Sanctum, Redis+Horizon, Reverb, Spatie Permission +
Activity Log + Laravel Data, Pest, PHPStan level 8, Pint, Docker
Compose, CI GitHub Actions. KONFIRMASI dulu daftar package yang akan
masuk composer.json sebelum install (aturan JANGAN). Belum ada fitur
— hanya kerangka.
Fase 1Skema, model, factory, enumdata
prompt
Buat migrasi + model + factory untuk semua entitas di
docs/BUSINESS-FLOW.md §4. Tambah kolom version (integer, default 1)
di appointments (optimistic lock) dan booked_count di slot_windows.
Buat enum: AppointmentStatus, MoveType, SlotWindowStatus,
GateTransactionType. Unique constraint (slot_window_id, container_no)
untuk appointment aktif. Belum ada business logic. Pakai skill
laravel-specialist + php-modernization.
Fase 2RBAC + Auth + Seeder jalandata
prompt
Implement Sanctum token auth (login/logout, issue token + abilities
per scope di docs/BUSINESS-FLOW.md §1). Pasang RolePermissionSeeder &
DemoSeeder (database/seeders) ke DatabaseSeeder. Buat Policy:
AppointmentPolicy (transporter hanya company_id sendiri; driver hanya
driver_id sendiri; planner reschedule/cancel hanya via
appointment.override + teraudit), Gate untuk gate-officer per
terminal_id. migrate:fresh --seed harus sukses. Pest test tiap aturan
Policy, termasuk override planner.
Fase 3Slot management (Planner)slice
prompt
Fitur Planner buka/tutup slot. OpenSlotWindowAction +
CloseSlotWindowAction (set status, BUKAN delete). GET
/api/v1/slots/availability pakai Cache::flexible([10,30]) +
Cache::tags(['slot',"gate:{id}"]). Event SlotWindowOpened →
invalidate cache + broadcast. Test: availability benar, cache
ter-invalidate saat window berubah.
Fase 4Booking — INTI, test-firstrawan
prompt
TEST DULU: Pest test race condition — 2 booking paralel ke slot
terakhir (capacity-1 terisi), pastikan hanya 1 sukses & 1 dapat 409
(DB::transaction + lockForUpdate). Lalu test Idempotency-Key (2
request key sama = 1 appointment). Setelah test MERAH, implement
BookAppointmentAction sesuai §3.2: lockForUpdate slot, cek
booked_count
Fase 5Reschedule / Cancel (optimistic lock)rawan
prompt
RescheduleAppointmentAction: terima version, lock window lama+baru
dalam SATU transaksi, cek version cocok (kalau tidak → 409), pindah
kuota, version++. CancelAppointmentAction: hanya sebelum ARRIVED,
kembalikan kuota. Path planner via appointment.override (teraudit).
Hormati state machine §2. Test: version basi ditolak; kuota balik
benar saat cancel; override planner tercatat di Activity Log.
Fase 6Gate-in / Gate-out (Officer)rawan
prompt
GateInAction & GateOutAction sesuai §3.5–3.6. Cek state machine +
terminal cocok + toleransi early/late dari config. Idempotency-Key di
endpoint gate. Event TruckGatedIn/Out → ProcessGateEventJob
(WithoutOverlapping per appointment, cek isGatedIn dulu) + broadcast
antrian. Test: gate-in pada status salah ditolak; gate event
idempoten.
Fase 7Jobs terjadwalbusiness
prompt
AppointmentReminderJob (H-2 jam, ShouldBeUnique uniqueId=appointment
id, set tries/backoff/failed). NoShowSweepJob (cron tiap 5 menit,
WithoutOverlapping, tandai NO_SHOW setelah window.end+grace,
kembalikan kuota dalam transaksi). Daftarkan di scheduler. Test pakai
time travel + Queue::fake.
Fase 8Realtime (Reverb + Echo)business
prompt
Reverb + Echo. Channel slot.{gateId} (sisa kuota live),
gate.queue.{terminalId} (antrian). Broadcast dari LISTENER event,
bukan dari Action. Private channel + otorisasi per role/company.
Fase 9Report + audithttp
prompt
Endpoint utilisasi (Planner): kuota vs terpakai vs no-show per
gate/tanggal. Pastikan Spatie Activity Log mencatat semua transisi
status, gate event, dan override planner. Transporter hanya lihat log
company sendiri (Policy).
Fase 10Frontend SPA (Vue 3)frontend
prompt
Vue 3 Composition API + TS + Pinia + Vue Router + TanStack Query +

lihat §07). View per role sesuai matriks §1: dispatcher
(availability+booking board), driver (jadwal hari ini + QR), gate
officer (scan + antrian), planner (slot config + utilisasi + override).
Vitest untuk komponen kritikal.
Aturan emas tiap fase: untuk Fase 4–6 (uang/kuota/konkurensi) test merah dulu, baru implementasi (§05). Tutup tiap fase dengan Definition of Done di §09 sebelum lanjut. Commit kecil per fase agar gampang rollback kalau melenceng.
05

Pola prompt: Plan dulu, eksekusi kemudian

// plan mode mencegah agent menulis kode yang melanggar kontrak

Untuk tugas non-sepele, masuk plan mode dulu (toggle Shift+Tab). Agent menyusun rencana tanpa menulis file — kamu review dulu apakah sesuai kontrak, baru setujui eksekusi. Ini titik kontrol terpentingmu.

prompt · plan mode — skema
Baca CLAUDE.md dan docs/BUSINESS-FLOW.md.

Susun rencana migrasi + model Eloquent untuk seluruh entitas
di §4, beserta relasi. Pastikan slot_windows punya kolom
capacity, booked_count, status; appointments punya version,
status, booking_code. Tampilkan rencana dulu — jangan tulis
file sampai saya setujui.

Setelah rencana oke, baru perintah eksekusi yang spesifik & ber-test:

prompt · eksekusi — Action inti (TDD)
Implementasikan BookAppointmentAction sesuai kontrak.

Wajib:
- Pest test DULU: simulasikan 2 booking pada slot dengan
  sisa kuota 1 → tepat 1 berhasil, 1 dapat 409.
- SlotWindow::lockForUpdate() dalam DB::transaction(attempts: 3).
- booked_count++ di transaksi yang sama. Tolak bila penuh /
  status != OPEN.
- Side-effect lewat event AppointmentBooked, BUKAN di Action.
- Idempotency-Key sudah ditangani middleware; jangan
  duplikasi di sini.

Jalankan composer test sampai hijau.

Pola promptnya selalu sama: rujuk kontrak → sebut invariannya (kuota, lock, version) → minta test sebagai bukti. Jangan biarkan agent memutuskan sendiri apakah perlu lock.

06

Subagent untuk truck-appointment-system

// isolasi konteks — main thread tetap ramping di tugas panjang

Subagent = instance Claude terpisah dengan konteks & izin tool sendiri; ia hanya mengembalikan hasil akhir, menjaga sesi utamamu bersih. Ada tipe bawaan — Explore (baca-saja, telusuri repo), Plan (merancang tanpa eksekusi), dan general-purpose — plus yang kamu definisikan di .claude/agents/.

Explore

"Telusuri semua Action & Repository yang menyentuh slot_windows" — tanpa membebani konteks utama dengan isi file.

Plan

Pasangan plan mode untuk fitur besar (mis. seluruh flow Reschedule lintas dua window).

Subagent kustom yang berharga untuk TAS — manfaatkan skill yang sudah ada di environment-mu:

.claude/agents/security-reviewer.md
---
name: security-reviewer
description: Audit tiap PR truck-appointment-system sebelum merge. Fokus auth
  Sanctum scope, Policy company_id/driver_id, mass-assignment,
  kebocoran data antar-tenant.
tools: Read, Grep, Glob
---
Gunakan skill laravel-security-audit. Periksa diff terhadap
OWASP & aturan RBAC di docs/BUSINESS-FLOW.md §1. Laporkan
temuan + lokasi file. Jangan ubah kode — review saja.
Trade-off nyata: subagent menyembunyikan konteks dari thread utama. Pakai saat isolasi memang berguna (review, eksplorasi paralel, test runner) — bukan untuk semua hal. Hingga beberapa subagent bisa jalan paralel; jangan over-pakai.

08

Menegakkan hardening (bukan sekadar berharap)

// jadikan tiap aturan kontrak sebagai test yang gagal kalau dilanggar

Aturan di kontrak hanya nyata kalau ada test yang merah saat dilanggar. Untuk TAS, empat invarian ini wajib punya test eksplisit — minta Claude Code menuliskannya sebagai bagian dari Definition of Done.

Race condition

2 booking konkuren ke slot sisa-1 → tepat 1 sukses. Tanpa test ini, lockForUpdate gampang "hilang" saat refactor.

Idempotency

2 POST dengan Idempotency-Key sama → 1 appointment, bukan 2.

Optimistic lock

Reschedule dengan version basi → 409, tidak menimpa perubahan orang lain.

Isolasi tenant

Transporter A akses appointment company B → 403. Driver lihat appointment bukan miliknya → 403.

prompt · guardrail test
Sebelum menandai fitur selesai, pastikan ada Pest test untuk
keempat invarian hardening yang relevan (race, idempotency,
optimistic lock, isolasi tenant). Tunjukkan test-nya, lalu
buktikan dengan composer test --parallel hijau.
09

Loop verifikasi tiap PR

// tiga perintah dari kontrakmu = gerbang kualitas

Kontrakmu sudah mendefinisikan perintahnya. Jadikan ini ritual penutup tiap slice — minta agent menjalankannya dan memperbaiki sampai bersih, bukan kamu yang mengejar.

terminal · gerbang sebelum commit
composer test      # pest --parallel
composer analyse   # phpstan level 8
composer fix       # pint

Lalu lepas security-reviewer (sec. 06) pada diff. Hanya setelah keempatnya hijau, baru rujuk balik ke CLAUDE.md bila ada keputusan baru yang perlu dicatat.

Definition of Done — sebuah fitur belum selesai sampai semua ini terpenuhi:

  • Tidak ada query/logika di controller (invokable → panggil Action → balik Resource).
  • Input lewat Form Request + DTO; output lewat Resource. Tidak ada Model::create($request->all()).
  • Endpoint di bawah /api/v1/; side-effect lewat Event/Listener, bukan di dalam Action.
  • Fitur kuota/gate: ada test race condition + idempotency yang hijau.
  • composer test hijau · analyse (PHPStan 8) bersih · fix jalan · security-audit lewat.
  • Policy ditegakkan & ada test-nya (cross-company / driver-self / gate-terminal / override-planner).
  • Activity Log mencatat perubahan status; CLAUDE.md masih ramping (detail baru → docs/).
Disiplin context: antar fitur besar, jalankan /clear untuk reset konteks percakapan. Kontrak tetap ter-load otomatis — yang dibuang hanya histori sesi yang sudah tak relevan, supaya kualitas tak menurun.
10

Jebakan umum

// hal yang bikin agent diam-diam melanggar kontrak
✕ "Buatkan semua endpoint sekaligus"

Konteks meledak, kualitas turun, hardening terlewat. → Satu vertical slice sampai test hijau, ulangi.

✕ Dispatch job di dalam DB::transaction()

Job bisa jalan sebelum commit → baca state basi. Kontrak sudah melarang; tegaskan di prompt Event/Listener.

✕ Menambah package diam-diam

Agent suka menarik library. Kontrakmu mewajibkan konfirmasi — pertahankan, dan cek composer.json di diff.

✕ CLAUDE.md membengkak

Detail panjang ke docs/. Bila kontrak lewat ~200 baris, ia mulai memakan konteks yang seharusnya untuk kode.

✕ Status appointment diubah lewat update() bebas

Lewati state machine = bug audit. Semua transisi lewat Action sesuai docs/BUSINESS-FLOW.md §2.

11

Sinkronisasi saat ada perubahan

// satu fakta hidup di satu tempat — sisanya mengikuti, tanpa drift

Lima artefakmu — CLAUDE.md, BUSINESS-FLOW.md, DUMMY-DATA.md, DemoSeeder.php, RolePermissionSeeder.php — saling bergantung. Kalau satu berubah sendirian, sisanya jadi bohong; dan seeder paling cepat ketahuan: migrate:fresh --seed langsung pecah. Aturan inti: tiap fakta punya satu sumber kebenaran, artefak lain hanya implementasinya.

Aturan & hardening

Sumber → CLAUDE.md. Implementasi → kode + test guardrail.

RBAC (role/akses)

Sumber → BUSINESS-FLOW §1. Implementasi → RolePermissionSeeder + Policy + Sanctum scope.

Skema / ERD

Sumber → BUSINESS-FLOW §4. Implementasi → migrasi + model + DemoSeeder.

State machine

Sumber → BUSINESS-FLOW §2. Implementasi → Action + status di DemoSeeder.

Scope MVP (kenapa)

Sumber → PRD.md §3. Implementasi → fitur yang dibangun; di luar daftar IN = tunda.

Peta propagasi — kalau yang berubah ini, maka wajib ikut update:

▸ Role / permission

BUSINESS-FLOW §1 (matriks + scope) → CLAUDE.md (Sanctum scope) → RolePermissionSeeder → Policy (company_id/driver_id) → test isolasi tenant.

▸ Entitas / kolom baru

BUSINESS-FLOW §4 → migrasi baru (jangan edit yang lama) → model → DemoSeeder → DUMMY-DATA.md → Resource + Repository (eager-load).

▸ Status / transisi baru

BUSINESS-FLOW §2 (state machine) → Action penegak transisi → DemoSeeder (tambah contoh appointment status itu) → DUMMY-DATA.md.

▸ Aturan hardening baru

CLAUDE.md → test guardrail yang merah saat aturan dilanggar.

▸ Endpoint / fitur baru

Cek PRD.md §3 dulu (in-scope?) → daftar Action di CLAUDE.md → proses di BUSINESS-FLOW → route /api/v1 → eksekusi via /slice. Kalau di luar scope, perbarui PRD lebih dulu.

Alur perubahan yang benar — dokumen dulu, kode mengikuti:

07 Commit ATOMIK + update status HANDOVER

Best practice anti-drift: (1) satu sumber kebenaran per fakta — jangan tulis matriks RBAC di dua tempat lalu lupa salah satunya. (2) PR atomik — dokumen, kode, seeder bergerak bersama. (3) Jaga migrate:fresh --seed selalu hijau di CI — itu menjadikan seeder sebagai test skema gratis. (4) Migrasi append-only di luar dev. (5) DemoSeeder harus selalu menyentuh semua status — kalau ada status baru di §2 tanpa contohnya di seeder, itu drift.
